SunSense Blinds System Overview

Project: SunSense Smart Blinds

Purpose: This document explains the current SunSense embedded system, its hardware connections, operating logic, and planned wireless integration so that new teammates can understand how the full product is intended to work.

1. System Summary

SunSense is an ESP32-S3 based smart blinds controller. It combines a light sensor, a rolling motor, a slat servo, a rotary encoder, a microphone, LEDs, and a temporary debug button to manage blinds automatically and manually.

The system has two main physical outputs:

  • Roll motor: raises and lowers the blinds.
  • Servo motor: rotates the slats between closed and open positions.

The main inputs are:

  • LDR module: determines whether it is bright or dark.
  • AS5600 encoder: reports blind position for endpoint detection and persistence.
  • INMP441 microphone: currently captures audio level data for future command support.
  • Debug button: currently used for local testing and mode changes, but not intended as the production user interface.

2. Product Direction

The production direction we have decided on is:

  • Wi-Fi + Home Assistant will be the primary user control path.
  • BLE will be used mainly for secure first-time provisioning and reprovisioning, not as the main day-to-day control channel.
  • Phone app / Home Assistant dashboard will act as the real user-facing controller.
  • Siri can later be supported through Home Assistant's HomeKit Bridge.
  • Android is also supported through the Home Assistant mobile app.
Important: The physical button is considered a development and service input. In production, the user is expected to interact primarily through their phone and home automation platform.

4. Mechanical Model

SunSense controls two different mechanical axes:

  • Blind travel: controlled by the gear motor, which rolls the blinds up and down.
  • Slat tilt: controlled by the servo, which rotates the slats.

The current slat convention is:

  • 0 degrees = slats fully closed
  • 90 degrees = slats fully open

The intended product home state is:

  • Blinds fully rolled down
  • Slats fully closed

5. Current Control Modes

AUTO Mode

In AUTO mode, the system uses the LDR to decide whether conditions are bright or dark.

  • If it is bright, the slats open to 90 degrees.
  • If it is dark, the slats close to 0 degrees.
  • The roll motor and slat servo are coordinated so that the servo moves first, and only after it reaches its target can the gear motor continue with the related blind movement.

MANUAL Mode

Manual mode is an override mode. It is intended for explicit user control, especially from a phone or home automation dashboard.

  • Manual mode currently overrides normal daytime automatic behavior.
  • However, if the environment becomes dark, the system is designed to override MANUAL and return to AUTO so the blinds can close.

This gives the user control during the day without allowing the system to remain open indefinitely after dark.

6. LDR Interpretation and Thresholds

The current LDR module behavior is:

  • Lower ADC value = brighter
  • Higher ADC value = darker

Current thresholds in firmware:

  • Bright threshold: 1200
  • Dark threshold: 1800

This means the firmware has hysteresis: it does not rapidly bounce between bright and dark around a single threshold.

7. Coordinated Motion Logic

One of the important recent design decisions is that the servo has higher logical priority than the roll motor.

That means the system does not treat these as two unrelated actions.

Instead, a typical open or close sequence works like this:

  1. Command the slats to the correct endpoint first.
  2. Wait for the servo to reach that endpoint.
  3. Only then allow the roll motor to move the blinds.

This sequencing is coordinated in software using FreeRTOS tasks plus central state logic in the main application.

8. Encoder and Position Tracking

The AS5600 encoder provides a position reading used for:

  • blind position estimation
  • endpoint detection
  • motor stopping
  • state persistence across power loss

The firmware also now uses a finite I2C timeout for encoder reads so the system is less likely to hang if the encoder is disconnected or miswired.

9. Runtime Persistence with NVS

One of the major reasons for choosing this ESP32 platform is NVS (non-volatile storage). SunSense is designed to remember meaningful runtime state even after power loss.

The runtime state layer stores values such as:

  • last known blind position
  • last operating mode
  • last light state
  • last slat angle target

On boot, the system restores this information and uses it as a recovery reference. If the encoder is healthy, live encoder data is preferred. If not, the last valid persisted state is used as the fallback estimate.

Recovery rule: the motor does not blindly resume motion on reboot. The system restores state, waits for a short startup settle window, and then allows AUTO behavior to continue if appropriate.

10. Wireless Architecture

Wi-Fi

Wi-Fi is the primary planned production connectivity path. The firmware includes an MQTT/Home Assistant integration scaffold so that SunSense can:

  • join the home network
  • connect to an MQTT broker
  • publish discovery data to Home Assistant
  • publish states such as mode, light level, motor state, and position
  • receive commands such as open, close, stop, and mode changes

This will allow the user to control the blinds from their phone, dashboards, and automations.

BLE

BLE is planned primarily for secure provisioning rather than normal operation.

The intended BLE provisioning model is:

  • device enters provisioning mode only on first boot or when explicitly triggered by a hidden service button
  • provisioning uses a unique per-device proof-of-possession secret
  • that secret is written during manufacturing and printed on a label or QR code
  • the device stores Wi-Fi and MQTT credentials into NVS after successful setup

This approach keeps provisioning secure while avoiding hardcoded shared setup credentials.

11. Home Assistant Integration Plan

The intended Home Assistant model is to expose SunSense as a smart blinds device rather than a loose collection of switches.

Planned entities include:

  • Cover entity: open, close, stop
  • Mode control: AUTO / MANUAL
  • Sensors: LDR raw, LDR filtered, light state, motor state, health, and position

Once exposed in Home Assistant, the device can also later be bridged into Apple Home for Siri support.

12. Planned Security Model

The current design direction includes four layers of security for provisioning and configuration:

  1. Provisioning is not always open.
  2. Reprovisioning requires an explicit physical trigger.
  3. Each unit has its own unique secret.
  4. Credentials are stored locally in NVS after setup.

Longer term, the design separates:

  • Factory data: device identity, serial, proof-of-possession secret
  • Network data: Wi-Fi and MQTT credentials
  • Runtime data: mode, position, light state, and slat state

13. Current Implementation Status

As of this document:

  • The core embedded architecture exists and builds cleanly.
  • LDR interpretation and thresholds have been corrected.
  • Servo logic is integrated with AUTO behavior.
  • Servo-first coordinated motion is implemented.
  • Manual mode now yields to darkness.
  • Runtime persistence has been added using NVS.
  • MQTT/Home Assistant integration scaffolding exists but still needs live broker and dashboard validation.
  • BLE provisioning is a planned next-stage feature and not yet fully implemented.

14. Key Open Next Steps

  1. Validate runtime persistence and reboot recovery on hardware.
  2. Finish Home Assistant MQTT discovery and end-to-end command testing.
  3. Implement secure BLE provisioning using per-device secrets.
  4. Implement full homing and factory-reset behavior, including the desired home state of blinds fully down and slats closed.
  5. Decide how manual slat control should be exposed in the future phone interface.
